## 1.==和hashCode和equals方法的区别和联系

==对于基本数据类型，==比较的是连个值是否相同

而对于引用类型（String，Object，自定义类型）比较的是是否是同一个对象，也就是比较的是地址



equals方法是Object类的一个方法，而Java中所有类都继承Object类，所以所有类都有equals方法，equals方法比较的是两个对象是否是同一个对象。

```java
public boolean equals(Object obj) {
    return (this == obj);
}
```

而对于一般的类/自定义类（除了String类），在没有重写equals方法时，使用equals方法比较的是两个对象是否相同，也就是是否是同一个对象。


2. 当两个String对象的内容也相同时（字面一样），也会返回true。

## Java序列化与反序列化

序列化：将数据结构和**对象转化为可以存储或传输（转为Json字符串保存到redis/发送给前端）**的形式，通常是**二进制字节流，也可以是JSON，XML**等文本格式

反序列化：将对象序列化后产生的数据转化为原始数据结构或对象的过程

序列化和反序列化常见应用场景：

- 对象在进行网络传输（比如远程方法调用 RPC 的时候）之前需要先被序列化，接收到序列化的对象之后需要再进行反序列化；
- 将对象存储到文件之前需要进行序列化，将对象从文件中读取出来需要进行反序列化；
- 将**对象存储到数据库（如 Redis）之前需要用到序列化，将对象从缓存数据库中读取出来需要反序列化**；
- 将**对象存储到内存之前需要进行序列化，从内存中读取出来之后需要进行反序列化**。

**总结：序列化的主要目的是通过网络传输对象或者说是将对象存储到文件系统、数据库、内存中**

## String，StringBuffer，StringBuilder的区别

String是不可变字符串（final char[]），在进行字符串拼接时，会重新创建字符串对象

StringBuffer和StringBuilder是可变字符串(char[] )，append()方法拼接字符串是在原字符数组上修改，只有在可变字符数组不够用的时候，才会扩容（原容量 ×2+2），不会产生无用的字符串对象

StringBuffer是线程安全的类，所有公开方法都是由synchronized修饰，然后效率低

StringBuilder是线程不安全的类，会出现线程安全问题，但是效率高

**1.HashMap的table数组容量为什么总是2的幂，如果不是会发生什么？**

- 只有当数组容量是2的幂时，数组长度-1才会出现低位都是1，那么在和hashCode进行&运算时才能够散列的均匀
- 数组容量总是2的幂是为了保证在扩容时，无需重新计算元素的哈希值，可直接定位元素在扩容后的数组中的位置

```java
    public static int disturbHashIdx(String key, int size) {
        return (size - 1) & (key.hashCode() ^ (key.hashCode() >>> 16));//hashCode与hashCode高16位异或^
    }
	//没有扰动函数的下标计算
    public static int hashIdx(String key, int size) {
        return (size - 1) & key.hashCode();
    }
}
```

## ArrayList和LinkedList

ArrayList底层数据结构是**数组**，既然是数组那么就是在**O(1)**的时间复杂度进行**随机访问**，在插入和删除元素时需要移动其他元素。

LinkedList底层是**双向链表**，所以**不支持随机访问**，时间复杂度是**O(n)**，在插入和删除元素时只需更改引用的指向即可，无需移动元素。节点不仅要存储数据，还要存储两个引用，所以更占内存。

## HashSet和TreeSet

两者都是set，所以其中中的元素不能重复，并且它们都是线程不安全的，二者的区别是：

1. HashSet中的元素可以是null，但TreeSet中的元素不能是null；
2. HashSet不能保证元素的排列顺序，而TreeSet支持自然排序、定制排序两种排序的方式；
3. **HashSet**底层是采用**哈希表**实现的，而**TreeSet**底层是采用**红黑树**实现的。



## HashMap和TreeMap

- HashMap底层是基于哈希表实现的，可以允许一个null的key和多个不为null的value
- TreeMap底层是基于红黑树实现的，不允许key为null，但是允许多个为null的value
- HashMap是无序的
- TreeMap是有序的

#### 为什么要进行两次判空？

第一次判空：如果没有第一次判空，在多线程的环境下，如果每一个调用getInstance()的线程都竞争一下这个锁，但是只有一个线程能获取到锁，其他线程就会进入阻塞状态，直到获取锁的线程释放锁，那么很多线程就都会阻塞在获取锁的位置，大大降低了程序的运行速度和系统的并发力度

#### 对象instance为什么要用volatile修饰

```java
public static Single getInstance(){
		if(instance == null){//                      1
			synchronized(Single.class){
				//第一次访问的时候，发现instance没有指向任何对象，这时实例化一个对象
```

**1.指令重排序导致出现未初始化的instance对象**

标号2的这行代码可以分解为三行伪代码，执行顺序为a,b,c，如果不加volatile关键字修饰instance，那么可能会a,c,b这样执行

```java
a. memory = allocate() //分配内存
b. ctorInstanc(memory) //初始化对象
c. instance = memory   //设置instance指向刚分配的地址
```

那么当一个线程执行到标号2这个位置，并且发生指令重排序，已经执行了a,c,但是b还未执行，那么此时如果有线程执行到标号1这个位置，那么此时instance就判空失败，直接返回一个未初始化的instance对象

**2.添加volatile关键字添加可见性**

当标号2这一行执行后，volatile instance的值被修改了，立刻刷新到主存中，其他线程中的instance缓存失效，需要冲主存中重新拷贝，也就是保证了其他线程的可见性



**注意事项**

- **线程安全**：`getInstance()` 方法中需要使用同步锁 `synchronized (Singleton.class)` 防止多线程同时进入造成实例被多次创建。
- **延迟初始化**：实例在第一次调用 `getInstance()` 方法时创建。
- **序列化和反序列化**：重写 `readResolve` 方法以确保反序列化时不会创建新的实例。
- **反射攻击**：在构造函数中添加防护代码，防止通过反射创建新实例。
- **类加载器问题**：注意复杂类加载环境可能导致的多个实例问题。

### 什么时候使用饿汉模式，什么时候使用懒汉模式？

饿汉模式：

是立即加载的方式，无论是否会用到这个对象，都会加载。
如果在构造方法里写了性能消耗较大，占时较久的代码，比如建立与数据库的连接，那么就会在启动的时候感觉稍微有些卡顿。

懒汉模式：

是延迟加载的方式，只有使用的时候才会加载。 并且有*线程安全*的考量。
使用懒汉式，在启动的时候，会感觉到比饿汉式略快，因为并没有做对象的实例化。 但是在第一次调用的时候，会进行实例化操作，感觉上就略慢。

看业务需求，如果业务上允许有比较充分的启动和初始化时间，就使用饿汉式，否则就使用懒汉式

## 2. 垃圾回收算法

### 如何判断对象已死？

**引用计数法：**每个对象有一个count属性，用来记录该对象被几个对象所引用（该方法不再使用，因为循环引用会出现问题）

**可达性分析：**

从一系列称为 GC Roots 根对象作为起始节点，如果所有的GC Roots都无法到达某个对象，说明该对象已死。

GC Roots 包括；

1. 全局性引用，对方法区的静态对象、常量对象的引用
2. 执行上下文，对 Java方法栈帧中的局部对象引用、对 JNI handles 对象引用
3. 已启动且未停止的 Java 线程

**两大问题**

1. 误报：已死亡对象被标记为存活，垃圾收集不到。多占用一会内存，影响较小。
2. 漏报：引用的对象（正在使用的）没有被标记为存活，被垃圾回收了。那么直接导致的就是JVM奔溃。（STW可以确保可达性分析法的准确性，避免漏报）





### 垃圾回收算法：

#### 标记—清除算法

![image-20250702154112739](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250702154112739.png)

- 标记无引用的死亡对象所占据的空闲内存，并记录到空闲列表中（free list）。
- 当需要创建新对象时，内存管理模块会从 free list 中寻找空闲内存，分配给新建的对象。
- 这种清理方式其实非常简单高效，但是也有一个问题**内存碎片化太严重了**。
- **Java 虚拟机的堆中对象**，必须是连续分布的，所以极端的情况下可能即使总剩余内存充足，但寻找连续内存分配效率低，或者严重到无法分配内存。*重启汤姆猫！*
- 在CMS中有此类算法的使用，GC暂停时间短，但存在算法缺陷。

总结一下就是：通过可达性分析来标记（多次标记）哪些对象已死，然后回收（对象会有一个自我拯救的过程），但是容易产生内存碎片，不易开辟连续的内存空间

**自我拯救**：当GC Roots无法与某个对象形成引用链，它将会被第一次标记并且进行一次筛选，筛选的条件是此对象是否有必要执行finalize()方法。当对象没有覆盖 finalize()方法或者finalize()方法已经被JVM调用过，虚拟机会将这两种情况都视为"没有必要执行"，此时的对象才 是真正"死"的对象。

#### 标记—复制算法

![image-20250702154823032](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250702154823032.png)

- 从图上看这回做完垃圾清理后连续的内存空间就大了。
- 这种方式是把内存区域分成两份，分别用两个指针 from 和 to 维护，并且只使用 from 指针指向的内存区域分配内存。
- 当发生垃圾回收时，则把存活对象复制到 to 指针指向的内存区域，并交换 from 与 to 指针。
- 它的好处很明显，就是**解决内存碎片化问题**。但也带来了其他问题，**堆空间浪费了一半**。

总结一下：就是将堆空间一分为二，只在其中一半的区域创建对象，当进行一次垃圾回收后，将存活的对象复制到没有使用的堆内存区域。解决了内存碎片的问题，但是同一时刻，堆区域只使用的一半。



#### 标记—压缩算法

![image-20250702155328228](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250702155328228.png)

- 标记的过程和标记清除算法一样，但在后续对象清理步骤中，先把存活对象都向内存空间一端移动，然后在清理掉其他内存空间。
- 这种算法能够解决内存碎片化问题，但压缩算法的性能开销也不小。

总结一下：解决了标记复制算法中堆内存只使用了一半的问题，但是**压缩算法的性能开销不小**

## 3. 类的加载过程

分为：加载，连接，初始化，使用，卸载

连接又分为：验证，准备，解析

**验证：验证是连接阶段的第一步，这一阶段的目的是确保 Class 文件的字节流中包含的信息符合《Java 虚拟机规范》的全部约束要求，保证这些信息被当作代码运行后不会危害虚拟机自身的安全。**

**准备：准备阶段是正式为类变量分配内存并设置类变量初始值的阶段**，这些内存都将在方法区中分配。

**解析：解析阶段是虚拟机将常量池内的符号引用替换为直接引用的过程。** 解析动作主要针对类或接口、字段、类方法、接口方法、方法类型、方法句柄和调用限定符 7 类符号引用进行。

**初始化：初始化阶段是执行初始化方法 `<clinit> ()`方法的过程，是类加载的最后一步，这一步 JVM 才开始真正执行类中定义的 Java 程序代码(字节码)。**

**类卸载：卸载类即该类的 Class 对象被 GC。**

卸载类需要满足 3 个要求:

1. 该类的所有的实例对象都已被 GC，也就是说堆不存在该类的实例对象。
2. 该类没有在其他任何地方被引用
3. 该类的类加载器的实例已被 GC



### 双亲委派模型的执行流程：

- 在类加载的时候，系统会首先判断当前类是否被加载过。已经被加载的类会直接返回，否则才会尝试加载（每个父类加载器都会走一遍这个流程）
- **类加载器在进行类加载的时候，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成（调用父加载器 `loadClass()`方法来加载类）**。这样的话，所有的请求最终都会传送到顶层的启动类加载器 **`BootstrapClassLoader`** 中。
- 只有当父加载器反馈自己无法完成这个加载请求（它的搜索范围中没有找到所需的类）时，子加载器才会尝试自己去加载（调用自己的 `findClass()` 方法来加载类）
- 如果子类加载器也无法加载这个类，那么它会抛出一个 `ClassNotFoundException` 异常

**JVM 判定两个 Java 类是否相同的具体规则**：JVM 不仅要看类的全名是否相同，还要看加载此类的类加载器是否一样。只有两者都相同的情况，才认为两个类是相同的。即使两个类来源于同一个 `Class` 文件，被同一个虚拟机加载，只要加载它们的类加载器不同，那这两个类就必定不相同。

**双亲委派的好处**

双亲委派模型是 Java 类加载机制的重要组成部分，它通过委派父加载器优先加载类的方式，实现了两个关键的安全目标：避免类的重复加载和防止核心 API 被篡改。

JVM 区分不同类的依据是类名加上加载该类的类加载器，即使类名相同，如果由不同的类加载器加载，也会被视为不同的类。 双亲委派模型确保核心类总是由 `BootstrapClassLoader` 加载，保证了核心类的唯一性。

## ThreadLocal是什么？

ThreadLocal是一种线程隔离机制，保证了在多线程的环境下对共享变量访问的一个安全性，ThreadLocal使用空间换时间的思想，在每一个线程中都有一个容器，存储共享变量的一个副本。那么每一个线程只对自己的变量副本进行更新操作，从而解决线程安全问题。具体实现原理是Thread类中都有一个ThreadLocalMap的变量存储









## 乐观锁和悲观锁

乐观锁：乐观锁**假定数据一般不会出现并发错误，只有在数据要提交更新的时候，才会对数据是否产生并发冲突进行检查**

悲观锁：总是假设最坏的情况，**每次去拿数据的时候都认为别人会修改，所以每次在拿数据的时候都会上锁**







## 并发编程的三个概念

原子性：一组操作要么全部执行，要么全部失败

可见性：一个线程修改了某个变量的值，其他线程能够立刻看到修改的值

有序性：程序执行的顺序按照代码的顺序执行，但是JVM内存模型允许一定的指令重排序

## synchronized

**1.原子性**

被synchronized修饰的类或者对象，在执行过程中必须先获得类或者对象的锁，直到执行完毕才会释放锁，这个过程保证了原子性

**注意！面试时经常会问比较synchronized和volatile，它们俩特性上最大的区别就在于原子性，volatile不具备原子性。**

**2.可见性**

可见性是指多个线程访问一个资源时，该资源的状态、值信息等对于其他线程都是可见的。

synchronized和volatile都具有可见性，其中synchronized对一个类或对象加锁时，一个线程如果要访问该类或对象必须先获得它的锁，而这个锁的状态对于其他任何线程都是可见的，并且在释放锁之前会将对变量的修改刷新到主存当中，保证资源变量的可见性，如果某个线程占用了该锁，其他线程就必须在锁池中等待锁的释放。

而volatile的实现类似，被volatile修饰的变量，每当值需要修改时都会立即更新主存，主存是共享的，所有线程可见，所以确保了其他线程读取到的变量永远是最新值，保证可见性。

**3.有序性**

**有序性值程序执行的顺序按照代码先后执行。**

synchronized和volatile都具有有序性，Java允许编译器和处理器对指令进行重排，但是指令重排并不会影响单线程的顺序，它影响的是多线程并发执行的顺序性。synchronized保证了每个时刻都只有一个线程访问同步代码块，也就确定了线程执行同步代码块是分先后顺序的，保证了有序性。

**4.可重入性**

synchronized和ReentrantLock都是可重入锁。当一个线程试图操作一个由其他线程持有的对象锁的临界资源时，将会处于阻塞状态，但当一个线程再次请求自己持有对象锁的临界资源时，这种情况属于重入锁。通俗一点讲就是说一个线程拥有了锁仍然还可以重复申请锁。

**synchronized可以修饰静态方法、成员函数，同时还可以直接定义代码块，但是归根结底它上锁的资源只有两类：一个是对象，一个是类。**

**synchronized底层实现**

在理解锁实现原理之前先了解一下Java的对象头和Monitor，在JVM中，对象是分成三部分存在的：对象头、实例数据、对其填充。

<img src="C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250526103554542.png" alt="image-20250526103554542" style="zoom:33%;" />

实例数据和对其填充与synchronized无关，**实例数据**存放类的属性信息，包括父类的属性信息，如果是数组实例数据还会存放数组的长度。**对其填充**不是必须部分，由于虚拟机要求对象起始地址必须是8字节的整数倍，对齐填充仅仅是为了使字节对齐。

**对象头**是我们需要关注的重点，它是synchronized实现锁的基础，因为synchronized申请锁、上锁、释放锁都与对象头有关。

对象头主要结构是由**`Mark Word`** 和 **`Class Metadata Address`**组成，其中**`Mark Word`**存储对象的hashCode、锁信息或分代年龄或GC标志等信息**，`Class Metadata Address`**是类型指针指向对象的类元数据，JVM通过该指针确定该对象是哪个类的实例。

锁的类型和状态在对象头**`Mark Word`**中都有记录，在申请锁、锁升级等过程中JVM都需要读取对象的**`Mark Word`**数据。

**每一个锁都对应一个monitor对象**，在HotSpot虚拟机中它是由ObjectMonitor实现的（C++实现）。每个对象都存在着一个monitor与之关联，对象与其monitor之间的关系有存在多种实现方式，如monitor可以与对象一起创建销毁或当线程试图获取对象锁时自动生成，但当一个monitor被某个线程持有后，它便处于锁定状态。

## volatile

**volatile的特性**

- 保证**可见性**，但不保证原子性：当写一个volatile变量时，JMM会把线程本地内存中的变量强制刷新到主内存中去，这个写操作会导致其他线程中的volatile变量缓存无效（**本地内存和主存是抽象的感念，并不一定就真实的对应cpu缓存和物理内存**）

- 禁止指令重排序：当程序执行到对volatile变量进行读或者写操作时，该读或者写操作之前的操作肯定全部进行，且结果对后面的操作可见；其后面的操作肯定还没有进行。**执行到volatile变量时，其前面的语句都执行完了，后面的语句都未执行，且前面执行的结果对后面可见。**

  ![image-20250818151914330](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250818151914330.png)

**不被volatile修饰的变量也会被写入到主内存中，但是写入的时机不确定，可能会导致其他线程无法看到最新的值（可见性问题）**

**所以volatile的作用是，当被volatile修饰的变量被修改时，会立即刷新到主内存中，其他线程本地的对于该变量的缓存将失效，保证了可见性**

**volatile的原理**

当加入volatile关键字后，相当于形成一个内存屏障，内存屏障确保

- 指令重排序不会把后面的指令重排序到内存屏障之前，也不会把前面的指令重排序到内存屏障后面；即在执行到内存屏障这句指令时，在他前面的操作已经全部完成；
- 强制将对**volatile修饰的变量修改操作立即写入主存**；
- 如果是写操作，它会导致其他CPU中对应的缓存行无效；

## 说说Session 和 Cookie 有什么联系和区别?

**Session 和 Cookie 到底有什么不同呢？**

- 存储位置不一样，**Cookie 保存在客户端**，**Session 保存在服务器端**。

- 有效期不同，**Cookie 可设置为长时间保持，比如我们经常使用的默认登录功能**，**Session 一般有效时间较短**，客户端关闭或者 Session 超时都会失效。
- 隐私策略不同，**Cookie 存储在客户端，比较容易遭到不法获取**，早期有人将用户的登录名和密码存储在 Cookie 中导致信息被窃取；S**ession 存储在服务端，安全性相对 Cookie 要好一些**。
- 存储大小不同， 单个Cookie保存的数据不能超过4K，Session可存储数据远高于 Cookie。

### 那么TCP为什么可靠

TCP的可靠传输体现在多个方面，如连接管理，校验和，确认应答，超时重传，滑动窗口，拥塞控制等

### 讲讲滑动窗口（流量控制）

TCP通过滑动窗口实现流量控制，发送方与接受方在进行数据传输的过程中，各自维护一个窗口，发送方发送数据给接收方，接收方回应ack的同时会告知发送方我还能接受多少数据（**通过TCP头部win字段来告知**），而接受方根据这个信息来决定数据发送的大小，当接收方告知发送方目前我的缓冲区已经满了，那么发送方就暂停发送数据，开启定时任务，发送试探报文看接收方是否能接受数据了

![image-20250819171246952](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250819171246952.png)

![image-20250819171329119](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250819171329119.png)



### 讲讲拥塞控制

发送方维护一个**拥塞窗口 cwnd（congestion window）** 的变量，调节所要发送数据的量

此时，发送窗口win的大小就变为了min（拥塞窗口大小，发送窗口大小），两者之间的最小值

因为计算机网络处于一个共享的状态，所以主机与主机之间的通信可能会使得网络拥堵

所以当网络没有发生拥堵时，拥塞窗口就会变大

当发生拥堵时，拥塞窗口就会变小

**常见的拥塞算法**

**慢启动算法**：

简单来说，慢启动算法就是每收到一个来此接收方的ack，拥塞窗口（cwnd）大小就会+1，所以慢慢的一次能够发送的数据就会增大，但是如果一味的增加必然会增加网络拥堵的几率，所以慢启动算法有一个**慢启动阈值**

**拥塞避免算法**：

当拥塞窗口大小cwnd达到了慢启动阈值就会进入拥塞避免算法，此时，拥塞窗口大小就会缓慢增长避免导致网络堵塞

**拥塞发生算法**：

当丢包时，发生超时重传（RTO）就会进入拥塞发生算法，此时拥塞窗口大小置为1，重新进入慢启动算法增加cwnd的大小

**快速恢复算法**：

快速恢复算法不会直接将cwnd置为1，因为它认为网络其实没有那么拥堵，所以只是将cwnd降低了一些

# MySQL篇

## 1.什么是事务？

事务是一组操作的集合，是一个不可分割的单位，事务中的操作要么全部成功，要么全部失败

## 2.事务的四大特性

**原子性**：事务时不可分割的最小单位，要么全部成功，要么全部失败

**一致性**：事务完成时，必须使所有的数据保持一致的状态

**隔离性**：数据库系统提供隔离机制，保证事务在不受外部并发操作影响的独立情况下运行

**持久性**：事务一旦提交或者回滚，对数据库中数据的改变是永久的

## 3.MySql默认的事务隔离级别

mysql事务的隔离级别有四种，分别是读未提交，读已提交，可重复读，串行化，而默认的是可重复读，在该隔离级别下会存在幻读的情况，即执行select语句时发现不存在id为8的数据，但是当我insert一条数据到id为8的这一行，发现插入失败（这是因为其他事务插入了该条数据）

**脏读**：当前事务读取到了另一个事务还未提交的数据

**不可重复读**：一个事务**先后读取同一条数据**，但两次读取的结果不同

**幻读**：一个事务**按照条件查询数据时**，没有对应的数据行，但是在插入数据时，又发现这行数据已经存在，好像出现了“幻影”

|                                     | 脏读 | 不可重复读 | 幻读 |
| ----------------------------------- | ---- | ---------- | ---- |
| Read uncommitted（读未提交）        | 1    | 1          | 1    |
| Read committed（读已提交）          | 0    | 1          | 1    |
| Repeatable Read（可重复读）（默认） | 0    | 0          | 1    |
| Serializable（串行化）              | 0    | 0          | 0    |

## 4.如何在可重复读的隔离级别下解决幻读的问题？

读分为两种读，一种是快照读，另一种是当前读

**先来说说快照读下是如何解决幻读的**

MVCC多版本并发控制结合ReadView和undo log版本链避免幻读的

在RR也就是可重复读事务的隔离级别下，一个简单的select语句也就是快照读会创建一个 Read View，该事务未提交前，后续的查询语句都会沿用该 Read View，而Read View中有几个核心字段

- 当前活跃事务id集合
- 最小活跃事务id
- 预分配事务id
- Read View创建的事务id

所以在快照读的时候，ReadView会沿用，所以在该事务未提交前所有的快照读查询的结果都是一致的，不会出现新的行（记录）



**mysql中除了普通查询之外都是当前读**，比如update，insert，delete这些语句在执行之前都要先查询最新版本的记录

事务A：select ... for update这样的查询就是**当前读，如果在范围查询情况下，比如select * from tb_user where id > 10 for update,那么就会在(2, +∞] 的 next-key lock（next-key lock 是间隙锁+记录锁的组合）范围加锁**，

事务B如果在该期间想要往这个范围插入记录，那么就会生成一个插入意向锁，只有**当事务A将锁释放（也就是事务A提交后），事务B才能插入记录**，这样就避免了由于事务B的插入操作导致事务A出现幻读现象







## 5.MVCC原理

MVCC是多版本并发控制，对一条记录会维护它的多个版本，通过 **ReadView** 和 **undo log 版本链**，以及**记录的隐藏字段**（记录事务id，表示哪一个事务对该记录进行了修改）实现对一条记录的维护

在不同的事务隔离级别下，ReadView视图的生成规则不同，在**RR也就是可重复读级别下**，在**第一次执行快照读的时候生成ReadView**，并且**后续快照读都将延续这个ReadView**，直到事务提交；**也就是以第一次快照读时当时数据库事务的提交状态来对比数据是否可见，所以就能够实现可重复读，对同一条记录的查询始终保持一致**。

而**RC读已提交隔离级别下**，**每一次快照读都会生成新的ReadView，也就是每次查询都会根据数据库当时事务的提交状态来对比数据的可见性，所以每一次查询都能查到已提交的事务的结果**。









## 说说InnoDB为什么选择B+树来存储索引（说说B+树）

B+非叶子节点只存储键也就是key，可以存储多个键，叶子节点存储key和实际的数据，这与B树不同，B树每一个节点既要存储键还要存储实际的数据；

B+树在查找的时候，在每一个节点内部都按照二分查找来查找，因为节点内部的key是有序的，因为非叶子节点不存储实际的数据，所以对于B+树而言，每一次查找都要走到叶子节点才能结束，而叶子节点之间通过指针连接，也就意味着，B+树支持范围查询 select * from table where id > 10

![image-20250801173535094](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250801173535094.png)

**为什么不选择红黑树，Hash，B树来存储索引呢？**

- 红黑树是二叉树，当数据量很大的时候，索引将会导致红黑树的层数很大，而索引是存储在磁盘上的，那么就意味着红黑树每向下查找一个节点就意味着要进行一次磁盘IO，这样效率就很低
- Hash，如果只搜索一个key的话，那么hash的效率还是很高的，但是hash存在几个问题，不支持范围查询，不支持索引值的排序操作，不支持联合索引的最左匹配原则
- B树，因为B树的查找可能会停止在任何一层，所以当范围查询的时候，需要做局部的中序遍历，在这个过程中可能会跨层访问，这就意味着会有更多的磁盘IO；同时因为B树每个节点不止要存储键还要存储实际的数据，所以相同的数据量下，B树的高度会更高，IO也就更频繁

所以选择B+树来存储索引，非叶子节点不存储数据只存储键，保证了树的高度不会太高，减少了IO操作，同时叶子节点之间用指针相连，所以支持范围查询。

## 讲一讲联合索引

联合索引是对多个列建立的索引，按照选择的列的顺序建立B+树，比如选择（name，age）两个字段建立联合索引（注意与age，name不同），先按照name字段排序，然后再按照age字段排序

在使用联合索引进行查询时，要符合最左前缀原则，查询要从索引列的最左列开始，否则索引有可能失效

## 1.redis String底层数据结构

两种：**SDS（动态字符串，simple dynamic string）和 整数**

当存储的是64位有符号整数，就会直接存储为long长整型

当存储的是非整数如"hello"，或着长度超过20字节的整数时，就会选择SDS作为底层结构

```c
struct sdshdr8 {
    uint8_t len;      // 已使用的字节长度（1字节，最大值255）
    uint8_t alloc;    // 总分配的字节长度（减去头部和终止符）
    unsigned char flags; // 标志位，标识 SDS 类型（如 sdshdr8）
    char buf[];       // 字符数组，存储字符串内容（以 \0 结尾，兼容C函数）
};
```

## 2.redis ZSet底层数据结构是什么？

**压缩列表ziplist**和**跳表skiplist+哈希表dict**

- **小数据量时使用：压缩列表**（member唯一）
- **大数据量时使用：跳表+哈希表**（member唯一）

**压缩列表是是一种连续内存存储结构**，所以会按照score+member这样一个一个存储，但是当存储的键值对多了，删除和插入需要频繁移动元素，适合小数据量存储

当数据量大了，就会使用**跳表skiplist+哈希表dict存储**

**跳表由多层有序链表组成，底层是完整存储键值对的有序链表，上层是底层的索引（每隔两个节点提取一个索引）**，而索引链表的节点中有三个域，一个down指针，指向下层索引节点或底层节点，一个next指针执行当前这一层的下一个索引节点，还有一个score分数域

![image-20250618160925595](C:\Users\Tibet\AppData\Roaming\Typora\typora-user-images\image-20250618160925595.png)

而哈希表dict是作为辅助结构的，用来快速定位某个member的score，所以哈希表存储的键值对是（member，score）

```java
ZSCORE key member1 //走哈希表 获得指定key中的member1的score
ZRANGE key start stop //走跳表，返回key中索引从start-stop区间内的member
```

### 问：如果zrange 一种有n个元素，取其中的m个元素时间复杂度是多少？

压缩列表：顺序遍历找到起始的位置，然后再遍历m个元素，所以时间复杂度O(n)

跳表+哈希表：先走跳表定位到起始位置logn，再顺序遍历m个元素，所以时间复杂度O(logn+m)

## Redis缓存一致性怎么解决（redis与mysql数据的一致性怎么解决）？

**主动更新+超时剔除（设置缓存的过期时间）**

针对读操作：缓存命中直接返回，未命中查数据库，然后写缓存并设过期时间

针对写操作：先写数据库，再删除缓存，同时要保证数据库与与缓存操作的原子性（**强一致性：利用数据库事务的原子性绑定数据库操作和缓存操作，两者有任何一个操作失败，就回滚重试，这样可能会影响性能，缓存长期不可用，业务就会阻塞**）

**如果追问为什么要先写数据库，再删除缓存呢？**

因为如果每一次写数据库都要更新缓存的话，那么在此期间，要是没有查询操作，那么更新缓存就是无用功了，所有只有在读的时候，再去写入缓存

**如果还追问强一致性的方法影响性能，有没有其他方法保证缓存一致性？**

**最终一致性：基于消息队列的最终一致性（异步补偿）**

利用消息队列的**可靠消息投递**和**重试机制**。保证缓存操作的执行成功，当有数据库变更时，通过发送mq的形式异步更新缓存，如果更新失败，还会重新发送mq，但可能会导致短时不一致（最终会一致），异步降低了主流程耗时，消息重试机制保证了缓存操作成功

## 2.RabbitMQ的工作方式有哪几种？

@RabbitListener 注解声明这是一个 RabbitMQ 消息监听器

四种

**基本模式**：一个生产者对应一个消费者，使用默认的Direct交换机，发送消息时直接指定队列的名称

**广播/扇出模式（Fanout）**：一个消息会被发送到所有绑定到该交换机的队列，用@QueueBinding注解来将队列绑定到交换机上

**路由模式（Direct）**：交换机会将消息路由到与 binding key 完全匹配的队列，这个binding key其实就相当于给队列重起了一个名字，不同的消费者可以监听同一个交换机的不同routing key（精确的路由键）

**通配符模式（Topic）**：使用通配符匹配 routing key，`*` 匹配一个单词，`#` 匹配零个或多个单词，典型应用：消息分类订阅
